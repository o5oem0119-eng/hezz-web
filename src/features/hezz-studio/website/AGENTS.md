# HEZZ Studio Website Agent Guide

Updated: 2026-07-21

## Project Root

Work inside `C:\Hezz company`. This directory is a HEZZ Studio subfeature, not an independent app or department. Do not create department-level package files, `node_modules`, or `.env` files.

## Responsibility

Own HEZZ Studio public web implementation, including the homepage, portfolio and campaign pages, Lab Notes, Guidebook and sales pages, approved checkout and delivery handoffs, disclosures, SEO, analytics instrumentation, domain and deployment preparation, release records, and maintenance.

## Canonical Documents

- `PUBLIC_WEBSITE_BRIEF.md`
- `PUBLIC_WEBSITE_COMPLETION_CHECKLIST.md`
- `WEBSITE_OPERATIONS.md`
- `docs/BEAUTY_CAMPAIGN_LAUNCH_PLAN.md` for approved campaign and product inputs

Do not duplicate full campaign plans inside website documents.

## Boundaries

- Advertising concepts, storyboards, and campaign edits belong to `../advertising`.
- Model profiles, visual references, and prompt systems belong to `../model-production`.
- Product, price, license, offer, and disclosure approval remain with the relevant HEZZ Studio or shared owner.
- Payment, order, delivery, credentials, and verified performance remain shared infrastructure.
- External client website production is outside the approved scope.

## Approval Points

Ask before changing shared route contracts, adding broad dependencies, activating analytics, connecting real credentials, changing a domain, enabling payment, deploying publicly, or publishing any campaign or product page.

## Forbidden Actions

- No automatic public deployment or domain mutation.
- No fake checkout success, clients, testimonials, or campaign results.
- No placeholder `#` links in a release build.
- No rejected or unapproved campaign assets.
- No direct edits to source campaign masters when producing web derivatives.
- No direct writes to long-term Obsidian knowledge unless mapped and approved.

Document shared contract changes in `docs/PROJECT_DECISIONS.md` and verify every affected consumer.
